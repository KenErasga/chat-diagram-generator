# Design Plan: Chat Diagram Generator

## 1. Problem Restatement and Deliverables

**Problem:** Build a split-screen web app where a user can describe diagrams in a chat interface, and the app generates and displays Mermaid.js flowcharts in response.

**Deliverables:**

- A React (NextJS) frontend with a chat panel and a diagram preview panel
- A NestJS backend that processes messages and returns diagram definitions or plain replies
- Stubbed LLM responses keyed on message content (no real API calls required)
- Multi-provider backend selectable via environment variable
- Conversation history owned by backend, keyed by `chatId`; frontend caches `chatId` and sends it per request

---

## 2. Architecture

/apps/
**chat-app** Frontend NextJS app (single page)

- Left panel: chat input
- Right panel: live Mermaid diagram render
- Communicates with backend over HTTP

**chat-service** Backend NestJS API server

- Single `/chat` endpoint
- Have a map of converstation history with an ID to keep track of (put it in as an adapter, might add a database in the future)
- Receives current message
- Selects provider stub via `MODEL_PROVIDER` env var
- Returns either a diagram definition or a plain text reply

**packages** Shared packages

- eslint config

---

## 3. Frontend (chat-app) vs Backend (chat-service) Responsibilities

**Frontend:**

- Generate and persist a `chatId` (e.g., in sessionStorage) on first load
- Send `chatId` + current message on every request
- Maintain a display-only message list in local state (text turns + latest diagram)
- Render Mermaid diagram from the latest diagram definition returned by backend
- Replace the displayed diagram on each new diagram response
- Display non-diagram replies as chat messages

**Backend:**

- Receive `chatId` and current message; look up conversation history from in-memory store
- Create a new history entry if `chatId` is not found
- Store history in an in-memory map (adapter pattern, database-ready for future)
- Route to the configured provider stub
- Decide whether to return a diagram (tool call) or a plain reply based on message content
- Append each turn (user + assistant) to the history entry
- Return a structured response indicating type: `diagram` or `message`

---

## 4. API Contract

**Endpoint:** `POST /chat`

**Request body:**

- `chatId`: string (UUID generated by frontend, cached in sessionStorage)
- `message`: the current user message (string)

Backend looks up history by `chatId`; creates a new entry if not found.

**Response body:**

- `type`: `"diagram"` or `"message"`
- `content`: plain text reply (always present)
- `diagram` (optional): Mermaid definition string, present only when `type` is `"diagram"`

---

## 5. Conversation Context

- Frontend generates a UUID `chatId` once per session and caches it in sessionStorage
- On every submission, the frontend sends only `chatId` + the new message
- Backend looks up the history entry for that `chatId` from its in-memory map
- Backend appends the user message and the assistant response to the history entry after each turn
- The provider stub receives the full accumulated history when deciding how to respond
- Frontend maintains a display-only list for rendering the chat UI — it does not own the source-of-truth history

---

## 6. Including Prior Diagrams in LLM Context

- When a diagram is generated, the backend stores the Mermaid definition alongside the assistant turn in the history entry
- On the next request, the backend retrieves the full history (including prior diagram definitions) from its in-memory store
- The provider stub (and any real provider) can read prior diagrams from that history to understand what was previously rendered
- This mirrors how a real LLM would receive diagram code as part of the assistant turn context

---

## 7. Stubbed Multi-Provider Backend

**Environment variable:** `MODEL_PROVIDER` (e.g., `openai`, `anthropic`, `stub`)

**Design:**

- A `ModelProvider` interface defines a single `chat(messages, currentMessage)` method returning the structured response
- Concrete stub implementations: `OpenAIStub`, `AnthropicStub`, `DefaultStub`
- A factory selects the implementation based on `MODEL_PROVIDER` at startup (registered via NestJS DI)
- All stubs share the same stub logic: if message contains "create" (case-insensitive), return a diagram; otherwise return a plain reply
- Swapping to a real provider later only requires implementing the interface and updating the factory

**Stub logic:**

- Message contains "create" (or similar keywords) → return a diagram tool call with a simple Mermaid flowchart
- Otherwise → return a conversational reply acknowledging the user's message

---

## 8. Diagram Rendering with Mermaid.js

- Install `mermaid` as a frontend dependency
- Render diagrams in a `<div>` using `mermaid.render()` or the `mermaid` React wrapper
- The diagram panel shows the most recently returned diagram definition
- On each new diagram response, replace the previous diagram entirely
- If the backend returns a `message` type (no diagram), the panel retains the last diagram
- Handle Mermaid parse errors gracefully — display an error state in the panel without crashing

---

## 9. Now vs Future

**Implement now:**

- Split-screen UI with chat and diagram panels
- `POST /chat` endpoint with `chatId`; backend manages history via in-memory adapter
- Mermaid.js diagram rendering
- Stubbed multi-provider backend controlled by env var
- Basic error handling (API errors, Mermaid parse errors)
- Unit tests for stub logic and API contract

**Leave for later:**

- Real LLM provider integrations (OpenAI, Anthropic, Bedroc)
- Streaming responses (SSE or WebSocket)
- Swap in-memory history adapter for a database-backed implementation
- Manual diagram editing
- Authentication
- Diagram export (PNG, SVG)
- Multiple diagram types beyond flowcharts

## 10. Assumptions, Constraints, Ambiguities, Trade-offs

**Assumptions:**

- One diagram displayed at a time; history only shows text turns in the chat panel
- The stub response quality is sufficient for demo/evaluation purposes
- No auth is required

**Constraints:**

- No real LLM API calls
- Frontend must be React
- Backend must be TypeScript
- Diagrams must use Mermaid.js

**Ambiguities:**

- Whether non-diagram replies should appear in the chat panel (assumed: yes)
- Whether the diagram panel should show a placeholder when no diagram has been generated yet (assumed: yes, show a prompt)

**Trade-offs:**

- In-memory history is simple but lost on restart — adapter pattern makes it easy to swap for a persistent store
- `chatId` in sessionStorage means history is lost on tab close; acceptable for this scope
- Stub keyed on "create" keyword is crude but easy to explain and extend
- No streaming keeps the implementation simple at the cost of perceived responsiveness

**Edge cases:**

- Very long conversation histories (no truncation strategy for now)
- Invalid Mermaid syntax returned by stub
- Backend unavailable — frontend should show an error in the chat panel
- Empty message submission — frontend should prevent this

**Reviewer expectations:**

- Clean separation of concerns between frontend and backend
- Typed API contract (shared types or clear interface)
- Stub logic is easy to replace with real provider
- Tests cover the core path (message → stub → response → render)

## 11. Test Plan

**Backend (NestJS + Jest):**

- Unit test each provider stub: given a message with "create", expect a diagram response; otherwise expect a plain reply
- Unit test the provider factory: given each `MODEL_PROVIDER` value, expect the correct stub to be returned
- Unit test the history adapter: new `chatId` creates a fresh entry; subsequent messages append correctly
- Integration test `POST /chat`: send a well-formed request, expect a valid structured response
- Integration test: two sequential `POST /chat` calls with the same `chatId` produce accumulated history

**Frontend (React Testing Library + Jest):**

- Test chat input: user types and submits a message, message appears in history
- Test diagram render: given a diagram response from a mocked API call, diagram panel updates
- Test error state: mocked API failure shows an error message in the UI
- Test no-diagram response: diagram panel retains previous diagram when response is plain text

**Manual smoke test:**

- Start both services locally
- Submit "Create a simple flowchart" — verify diagram appears
- Submit "What does this diagram show?" — verify plain reply, diagram unchanged
- Refresh page — verify state resets cleanly
